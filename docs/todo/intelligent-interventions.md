# Умные самостоятельные вмешательства бота

## Статус

Часть `v1` реализуется в плане [`../superpowers/plans/2026-04-11-intelligent-interventions-v1.md`](../superpowers/plans/2026-04-11-intelligent-interventions-v1.md).

В `v1` реализуется conservative path: входящее non-direct group-сообщение проходит cheap probability/cooldown gate, затем structured LLM-анализ окна, затем persona reply generation только если `shouldIntervene = true`. Таймерный сценарий `silence_after_activity` остаётся отдельным следующим шагом.

## Зачем это нужно

Сейчас бот всегда отвечает на `mention` и `reply`, а без прямого триггера может только случайно влезть в разговор по вероятности и `cooldown`.

Нужен более умный механизм, при котором бот сам решает, стоит ли ему сейчас вмешиваться в разговор, если его напрямую не звали.

Примеры желаемого поведения:

- если в чате стало тихо, бот может сам что-то вбросить, чтобы оживить разговор;
- если участники спорят, бот может решить, стоит ли ему поддержать динамику, подзадорить их или наоборот снизить напряжение;
- конкретная манера реакции должна зависеть от персонажа, а не быть жёстко зашитой в детектор ситуации.

## Уже согласованные правила

- На `mention` бот отвечает всегда.
- На `reply` бот отвечает всегда.
- Новый механизм нужен только для самостоятельных вмешательств без прямого вызова.

## Выбранный архитектурный подход

Выбран вариант `structured intervention analysis`.

Это означает:

- сначала отдельный слой оценивает ситуацию в чате в сухих, структурированных терминах;
- потом эта аналитика передаётся в persona-driven reply generation;
- уже персона, получив аналитику, решает, как именно реагировать.

## Архитектурный принцип

Это не локальное правило только для этой задачи, а один из базовых архитектурных принципов всего проекта:

- всегда отделять сухую аналитику ситуации от персоны;
- персона не должна определять, что происходит в чате;
- персона должна определять, как реагировать на уже описанную ситуацию.
- самостоятельные вмешательства должны работать по принципу `fresh-or-drop`: если аналитика устарела, лучше не отвечать вовсе.

Пример:

- аналитика определяет, что в чате `conflict` и рекомендуемая цель вмешательства `deescalate`;
- добрая персона постарается помирить участников;
- более дерзкая персона может реагировать жёстче, но всё равно в рамках зафиксированной аналитики ситуации.

## Структура решения

Предварительно у decision-слоя должен быть примерно такой результат:

```json
{
  "shouldIntervene": true,
  "situationKind": "debate",
  "goal": "provoke",
  "intensity": "medium",
  "reason": "participants are actively arguing but still engaged",
  "confidence": 0.77
}
```

Этот результат описывает ситуацию и цель вмешательства, но не готовую реплику. Текст ответа генерируется только на следующем шаге, уже с участием persona-слоя.

## Базовые цели вмешательства

На текущем этапе хотим заложить такие цели:

- `engage`
- `deescalate`
- `provoke`
- `joke`
- `support`

Их смысл:

- `engage` — оживить затухающий разговор;
- `deescalate` — снизить напряжение;
- `provoke` — подзадорить и усилить динамику;
- `joke` — разрядить или украсить разговор шуткой;
- `support` — поддержать участника или сторону в разговоре.

## Когда дёргать LLM-анализатор ситуации

Предварительная рекомендация:

- не отправлять в LLM каждый новый месседж;
- сначала прогонять локальные дешёвые проверки;
- только если чат выглядит как кандидат на самостоятельное вмешательство, запускать LLM-анализ.
- использовать один общий LLM-анализатор ситуации, а не отдельные LLM-вызовы под каждый частный сценарий.

### Локальные дешёвые проверки перед LLM

- это не `private`-чат;
- это не `mention` и не `reply`;
- бот сам не писал слишком недавно;
- накопилось хотя бы минимальное окно новых сообщений;
- чат не находится в состоянии, где вмешательство явно запрещено локальными правилами;
- по текущему состоянию job-coordinator можно безопасно запускать ещё одну оценку.

### Типовые поводы вызвать LLM-анализ

- в чате наступила заметная пауза после живого обсуждения;
- накопился короткий, но насыщенный обмен между несколькими участниками;
- появились признаки спора, подколов или эмоционального напряжения;
- разговор затухает, но ещё есть контекст, в который можно естественно встроиться.

### Предварительная идея по частоте

На старте лучше делать консервативно:

- не чаще одного анализа на чат за ограниченный интервал времени;
- не запускать анализ на каждое сообщение подряд;
- запускать анализ только после прохождения cheap gate и только когда появился новый meaningful context window.

Это нужно, чтобы:

- не сжигать токены без пользы;
- не делать поведение бота слишком суетливым;
- сохранить читаемую и объяснимую архитектуру.

## Как понимать стадию разговора

Для этой функции не нужно пытаться определять абсолютные `начало / середина / конец` всего диалога.

Вместо этого бот должен оценивать локальную фазу текущего эпизода разговора.

Предварительно полезны такие состояния:

- `warmup` — разговор только завёлся;
- `active` — идёт живой обмен;
- `debate` или `conflict` — есть спор, подколы или напряжение;
- `tapering` — разговор затухает;
- `silence_after_activity` — после движухи наступила пауза.

Эти состояния должны определяться не "чутьём персонажа", а по сухим сигналам, например:

- сколько новых сообщений в текущем окне;
- сколько уникальных участников в окне;
- как быстро идут сообщения;
- был ли заметный временной разрыв;
- растёт темп разговора или падает;
- продолжают ли последние сообщения один и тот же эпизод общения.

## Анализировать не сообщения, а окна контекста

Для `v1` не стоит запускать аналитику на каждый новый месседж.

Предпочтительная модель:

- каждое новое сообщение обновляет локальное состояние чата;
- анализ запускается только когда сформировался кандидат на вмешательство;
- анализ привязывается к конкретному окну сообщений;
- окно может быть вызвано паузой после активности или пакетным накоплением нового контекста.

Для разных сценариев это выглядит так:

- `silence_after_activity` — можно проверять по таймеру после периода живого разговора;
- `debate/conflict` — можно проверять после небольшого пакета новых сообщений или короткого активного интервала, а не после каждого отдельного месседжа;
- `tapering` — это тоже часть общего анализа ситуации, а не отдельный специальный вызов в LLM.

## Защита от опоздавших вмешательств

Если участники активно спамят сообщениями, бот не должен отправлять реплику в уже умерший или изменившийся контекст.

Поэтому в архитектуре нужно предусмотреть:

- одновременно не более одного `intervention-analysis` job на чат;
- новые сообщения во время анализа не запускают бесконечные параллельные проверки;
- чат может быть помечен как `needs_recheck`, если во время анализа пришёл новый значимый контекст;
- результат анализа должен быть привязан к конкретной верхней границе окна, например `analyzed_through_message_id`;
- перед отправкой самостоятельной реплики нужно проверять, не устарело ли решение;
- если решение устарело, вмешательство должно быть молча отброшено.

## Fresh-Or-Drop

Это отдельный ключевой принцип поведения:

- лучше иногда промолчать, чем вмешаться в уже ушедшую сцену разговора;
- если за время анализа контекст существенно изменился, бот не должен "догонять" чат репликой из прошлого состояния;
- самостоятельное вмешательство допустимо только если анализ ещё свежий и относится к текущему эпизоду общения.

## Что должно уходить в LLM на этапе аналитики

Не persona-реплика, а сухой контекст ситуации, например:

- последние `N` сообщений;
- число активных участников в окне;
- время с последнего сообщения бота;
- время с последней активности людей;
- признаки тишины, быстрого обмена, спора, напряжения;
- предварительная локальная фаза эпизода разговора, если она уже вычислена детерминированно;
- краткий chat summary;
- при необходимости краткие социальные сигналы из памяти.

## Что должно уходить в LLM на этапе ответа

На этапе генерации реплики в LLM должны уходить:

- persona;
- chat summary;
- participant memory без долгосрочной self-memory бота;
- participant/social context;
- recent messages;
- структурированная аналитика ситуации;
- причина, почему бот решил вмешаться именно сейчас.

## Что хочется сохранить на уровне логов

Так как выбран structured-подход, по логам должно быть видно:

- почему локальный gate вообще допустил запуск LLM-анализа;
- какое окно сообщений ушло в анализ;
- что вернул decision-слой: `shouldIntervene`, `situationKind`, `goal`, `intensity`, `confidence`;
- был ли ответ отправлен или отброшен как устаревший;
- если ответ не был отправлен, то по какой причине.

## Что пока не зафиксировано окончательно

- точный список `situationKind`;
- точный формат cheap local gate;
- минимальный размер окна сообщений для анализа;
- конкретный cooldown между двумя LLM-анализами по одному чату;
- критерии, по которым `provoke` допустим, а по которым уже лучше `deescalate`.
